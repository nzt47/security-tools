# 最危险的告警是"从不告警"——一次 Prometheus 静默失效的排查笔记

> **分享类型**: 监控体系 / 事故复盘  
> **关键词**: Prometheus、静默失效、兜底机制、指标命名  
> **阅读时间**: 约 6 分钟  
> **关联代码**: commit `1148538f`

---

## TL;DR

一个指标名单复数错误（`yunshu_http_requests_total` → `yunshu_http_request_total`），让所有 HTTP 错误率告警形同虚设。但因为 `or on() vector(0)` 兜底机制的存在，错误率"看起来"永远是 0，健康度评分"看起来"永远接近满分。**最危险的告警不是误报，而是从不告警。**

---

## 一、故事是怎么开始的

起因很普通：验证告警链路是否通畅。

服务启动后，我往 `/api/test/error` 端点灌了 40 次 5xx 请求，然后坐等 Prometheus 告警触发。结果——什么都没发生。

不是告警延迟，不是阈值没到，是**指标根本没采集到**。但奇怪的是，健康度仪表盘显示 stability_score = 100，overall_score = 86，一切"看起来"很健康。

这就是问题的开端：**一个本该报错的系统，表现得毫无破绽。**

---

## 二、"一切正常"的假象

先看几个关键指标当时的值：

| 指标 | 期望值 | 实际值 |
|------|--------|--------|
| `sum(yunshu_http_requests_total{status=~"5.."})` | 40 | **空** |
| `yunshu:health:stability:error_rate` | ~7% | **0** |
| `stability_score` | 应该下降 | **100** |
| YunshuHighErrorRate 告警 | 应该触发 | **从不触发** |

error_rate 永远是 0，健康度评分永远满分，告警永远不响。如果没有人主动去注入错误测试，这个问题可能还会潜伏更久。

**这就是"静默失效"最可怕的地方——它不像崩溃那样大张旗鼓，而是悄无声息地让监控系统变成摆设。**

---

## 三、顺藤摸瓜

排查的第一步是确认 Prometheus 到底抓到了什么指标。写了个脚本 `find_http_metrics.py` 列出所有 `yunshu_*` 开头的指标名，结果一目了然：

```
实际暴露的指标: yunshu_http_request_total       ← 单数
规则文件引用的: yunshu_http_requests_total      ← 复数
```

**一个 "s" 的差异，让所有引用这个指标的录制规则和告警规则全部失效。**

但这里有个关键问题：既然指标名错了，为什么录制规则没有报错，反而返回了 0？

答案在录制规则的表达式里：

```promql
(
  sum(rate(yunshu_http_requests_total{status=~"5.."}[5m]))
  /
  clamp_min(sum(rate(yunshu_http_requests_total[5m])), 1e-9)
) or on() vector(0)
```

最后的 `or on() vector(0)` 是"兜底机制"——当主查询返回空向量时，用 0 作为默认值。这本意是好的：防止除零错误，保证整体健康度评分计算不中断。但副作用是：**"指标不存在"和"指标值确实是 0"变得无法区分。**

---

## 四、真相大白

根因链路非常清晰：

```
prometheus_flask_exporter 注册指标时用单数 (yunshu_http_request_total)
        ↓
规则文件里全部写成复数 (yunshu_http_requests_total)
        ↓
PromQL 查询返回空向量（指标不存在）
        ↓
or on() vector(0) 兜底为 0
        ↓
error_rate 永远是 0
        ↓
stability_score 永远是 100
        ↓
告警条件 > 0.05 永远不满足
        ↓
告警从不触发，且无人察觉
```

修复很简单：把 3 个规则文件里的 11 处 `yunshu_http_requests_total` 改成 `yunshu_http_request_total`，reload Prometheus，等 30 秒。修复后重新注入 40 次 5xx，error_rate 立刻升到 7.51%，stability_score 降到 50，两条告警正确触发。

## 五、四条可以带走的知识

### 1. 兜底机制是"好仆人，坏主人"

`or on() vector(0)` 这类兜底逻辑能保证计算链不中断，但它会让"数据缺失"伪装成"数据为零"。**兜底应该配合可观测性**——当兜底值被使用时，应该有日志或告警提醒"嘿，主查询返回空了"，而不是默默吞掉问题。

**行动建议**：为关键录制规则增加"指标存在性告警"，当主查询持续返回空、仅靠兜底值运行时触发提醒。

### 2. 指标命名要对齐"事实"，而非"直觉"

`requests` 是复数，读起来更自然，但 `prometheus_flask_exporter` 生成的 counter 名是单数 `request_total`。**指标名的唯一事实来源是 `/metrics` 端点的实际输出**，不是规则文件、不是文档、更不是直觉。

**行动建议**：把指标名校验集成到 CI——每次规则文件变更时，自动校验引用的指标名是否在 `/metrics` 端点真实存在。本次排查用的 `find_http_metrics.py` 就可以直接改造。

### 3. "全部满分"反而要警惕

健康度评分 6 个维度，5 个 100 分、1 个 30 分（availability），整体 86 分——看起来很合理，对吧？但 stability 永远 100 分是因为 error_rate 永远 0，error_rate 永远 0 是因为指标名写错了。**当一个维度长期保持满分且毫无波动时，首先要怀疑的不是"系统真稳"，而是"数据真没了"。**

**行动建议**：定期审视健康度评分的趋势曲线。如果某条线是一条笔直的水平线（无论是 100 还是 0），大概率有问题。

### 4. 告警"既不误报也不漏报"是最危险的状态

误报会让人烦，但至少说明告警系统"活着"。漏报会让人慌，但至少出事时能意识到"告警没响"。**最危险的是"静默失效"——告警系统看起来在正常工作（规则加载成功、health=ok），但实际上对所有异常视而不见。** 这种状态下，人会逐渐信任一个已经失效的监控系统，真正的灾难来临时毫无防备。

**行动建议**：定期做错误注入演练（chaos engineering）。不是验证系统能不能扛住故障，而是验证**告警系统能不能发现故障**。本次的 `test_alert_chain.py` 就是一个最小化的告警链路验证脚本。

## 七、一句反思

监控系统的最大敌人不是复杂的架构，而是**"看起来正常"的假象**。兜底机制、默认值、静默降级——这些提升健壮性的设计，如果没有配套的可观测性，就会从"保护"变成"蒙蔽"。

下次当你看到一个永远满分、永远不报错的仪表盘，先问一句：**这是真的没问题，还是有问题但看不到？**
