## P6 具体执行任务

### 1. 编写全系统自动化测试集

在 `tests/` 目录下编写覆盖全系统的测试用例，必须包含：
- 单元测试（`tests/unit/`）
- 集成测试（`tests/integration/`）
- 功能测试（`tests/functional/`）
- 性能测试（`tests/performance/`）
- 安全测试（`tests/security/`）：对抗样本防御测试、偏见检测测试
- 工作流回归测试

### 2. 执行离线 E2E（端到端）测试

编写并执行在完全断网状态下的 E2E 测试脚本。验证全链路闭环：
- UI 输入 → 触发工作流 → 调用本地 LLM → 写入本地记忆 → 最终 UI 输出结果

### 3. 输出最终质量审计报告

汇总所有测试结果，输出最终的全面质量审计报告，包含：
- 日志摘要
- 测试结果分析
- 覆盖率统计
- 问题清单（含优先级）
- 修复验证结果
- 未覆盖部分的风险评估说明

### 4. 可解释性验证

基于 LIME 等工具进行可解释性验证。

## 📎 设计参考：全面测试与质量保障（第八部分 8.5 节）

> **以下设计规范为 P6 阶段的核心指导原则，执行时必须严格遵循。**

### 8.5 全面测试与质量保障

**评测即代码（Eval as Code）**：将测试用例和评估基准（Benchmark）内建到 CI/CD 流水线中。每一次代码提交或模型更新，都必须通过自动化回归测试。

**多维度验证**：除了常规的单元测试和集成测试，还需针对 AI 系统引入专项测试：
- 对抗样本防御测试（鲁棒性）
- 偏见检测测试（公平性）
- 基于 LIME 等工具的可解释性验证

**E2E（端到端）测试**：模拟真实用户请求，从 UI 输入 → 触发工作流 → 调用本地 LLM → 写入本地记忆 → 最终 UI 输出结果，验证整个闭环在离线状态下是否完全通畅。

## 📎 设计参考：资源与成本优化（第七部分）

> **以下设计规范为 P6 阶段性能测试和成本评估的参考依据。**

### 7.1 动态路由与模型降级

系统必须具备成本感知能力。
- 主 Agent 在拆解任务时，应优先将简单子任务路由给小模型（如 7B/8B 模型）。
- 只有在遇到复杂推理时才调用昂贵的旗舰模型。

### 7.2 上下文压缩与缓存（Semantic Caching）

- 对于重复或高度相似的用户请求，系统应直接命中缓存返回结果。
- 在分身之间传递信息时，不要传递原始长文本，而是传递由小模型提取的"结构化摘要"，大幅降低通信成本。
