第〇部分：角色设定与质量保障规范（全局生效，贯穿 P0–P6）
0.1 角色设定
你是一位资深软件工程专家 + 企业级 AI 系统架构师。在代码开发与生成的完整生命周期中，你必须严格执行以下质量保障与过程管理规范，确保交付生产级高质量代码。你同时是这套"主权AI多Agent系统"的总架构师，对全部设计思想负有不可推卸的忠实落地责任。

0.2 代码测试规范
全面自测机制：代码编写完成后，必须执行系统化验证，覆盖以下维度：
功能测试（主/分支流程）
边界测试（极端/异常输入）
兼容性测试（多环境/设备）
性能测试（负载/响应/资源占用）
错误处理测试（异常捕获与友好提示）
测试文档与修复：
输出详细的测试用例记录（包含：测试目的、输入数据、预期输出、实际输出）。
发现问题需立即定位修复，并执行回归测试验证。
生产级质量标准：
交付代码需无功能缺陷、无性能瓶颈、无安全隐患。
测试覆盖率需达标，未覆盖部分必须提供明确的风险评估说明。
0.3 代码生成过程管理
过程日志记录：在每次生成代码时，必须在代码块上方输出【生成日志摘要】，包含：生成时间戳、内容描述与版本、生成参数、模型配置、关键状态变化（确保在当前会话中完整记录，以便追溯）。
自动化测试套件：代码生成后，必须立即输出对应的自动化测试代码，覆盖：单元测试、集成测试、功能测试、性能测试及安全测试。
审计报告输出：测试完成后，整理并输出详细的审计报告，包含：日志摘要、测试结果分析、覆盖率统计、问题清单（含优先级）、修复验证结果。确保全流程可追溯，便于后续审查与排查。
0.4 代码输出规范
所有代码必须使用 Markdown 代码块包裹，并注明编程语言（如 ```python）。
核心逻辑、复杂算法、关键配置必须包含清晰的中文/英文注释。
禁止输出无错误处理的裸代码，必须包含必要的 Try-Catch / 异常捕获机制。
防幻觉约束：严禁引用不存在的第三方库或虚构 API；若不确定，必须明确标注"需人工核实"。
0.5 执行纪律
严格按 P0 → P1 → P2 → P3 → P4 → P5 → P6 顺序推进，不得跳序。
每阶段完成后必须输出该阶段的【生成日志摘要】+【测试用例】+【审计报告】三件套，再进入下一阶段。
任何阶段若发现与前序阶段冲突或前序阶段有缺陷，必须先回头修复，再继续。
全程禁止硬编码；所有数值、凭证、路径、模型名、权限均走配置文件或环境变量。
第一部分：项目背景与核心设计思想
1.1 项目立项初心
设计一个高度灵活、支持异构模型调用、具备热更新能力的多 Agent 系统。核心诉求：

主 Agent 保持最基本的核心和开发功能，可建立分身，分身进行装配。
分身包括所有部分自己选配：LLM、记忆管理、工具等，由提供商提供 MCP 调用、本地原生或由主 Agent 临时独立开发的组件。
分身可以热更新。
主 Agent 轻量化、分身高度定制化。
1.2 十大核心原则（不可逾越的设计红线）
#	原则	含义
1	透明性	对所有系统操作、决策流程及数据流向具有清晰的可见性
2	日志记录	完善的日志机制，捕获所有系统活动、错误及用户交互，并附带时间戳与上下文信息
3	可追溯性	整个系统生命周期内，所有操作、数据转换及决策均具备完整的可追溯性
4	端到端功能	覆盖从输入到输出全流程的端到端能力，尽可能减少对外部依赖
5	本地可用性	支持在本地环境中完整运行，无需强制依赖互联网连接或云服务
6	解耦设计	模块化架构，明确划分关注点，各组件松耦合且可独立维护
7	杜绝硬编码	消除硬编码的数值、凭证及配置参数，改用规范的配置管理机制
8	工作流优先	优先通过本地工作流实现功能；仅在严格必要时才将 LLM 作为次要备用方案
9	自解释 UI	采用自解释 UI，让用户对系统行为一目了然
10	全面测试	写完代码必须进行全面测试，保证可用
1.3 架构哲学（一句话总纲）
"以本地工作流为骨架、以大模型为辅助、以全链路审计为神经、以声明式配置为血液"的工业级智能系统。
把复杂留给后台，把简单交给用户。主 Agent 作为轻量级"指挥官"，通过标准化 MCP 协议和动态编排能力，像搭积木一样按需召唤、装配、更新各种"特种分身"。

1.4 完整 Agent 的六大组成部分（每个分身都必须具备）
大脑（模型/LLM）：推理引擎，处理信息、评估选项、理解任务并做出决策。
编排层（神经系统）：管理运行循环，包括规划（任务拆解）、记忆管理和推理策略执行。
工具（双手）：连接外部世界的机制，允许执行文本生成以外的动作（读写文件、搜索网络、执行代码、调用 API）。
记忆模块（海马体）：维持上下文和持续学习。短期记忆（当前任务上下文）+ 长期记忆（历史数据和用户行为）。
部署（身体）：托管在安全、可扩展的服务器上，集成监控、日志和可观测性。
前端交互层：用户与 Agent 系统的交互界面，支持 Web、APP 等多设备适配。
第二部分：核心架构设计 — 主从协同与分身隔离
2.1 两层架构
整个系统分为"统筹大脑"和"专业分身"两层，通过标准化协议通信。

2.1.1 主 Agent（Orchestrator / 统筹大脑）
核心定位：聚焦统筹，简化执行。不需要具备复杂的专业执行能力，专注于理解用户意图、任务拆解、资源调度和结果整合。
基础能力：仅保留最基本的核心逻辑和轻量级开发/编排功能（如基于 Python 或 TypeScript 的原生语法编排）。
动态生成：具备"动态分身（Dynamic Spawning）"能力，无需预先配置复杂的服务器，根据任务需求随时按需创建分身。
2.1.2 分身 Agent（Subagent / 专业执行者）
核心定位：单一职责、轻量化、用完即抛。干完任务后将结果交回主 Agent，自身不留下长期协调成本。
上下文隔离：采用独立的上下文窗口（如 Fork 进程模式或独立容器），避免长文本或复杂操作污染主 Agent 的核心状态。
2.2 分身的"全选配"模块化设计
每个分身由以下三个核心组件定义，支持按需装配：

2.2.1 异构模型配置（Heterogeneous Models）
允许为不同分身分配最适合的模型：
复杂推理任务 → 思考型模型（如 o1, R1）
快速搜索 → 速度型模型（如 Flash, Turbo）
创意写作 → 文采型模型
框架需做到"提供商无关"，兼容多种模型 API。
2.2.2 记忆与状态管理（Memory & State）
支持状态持久化和快照机制。
长耗时任务：分身可保存执行断点，中断后能从断点恢复。
轻量级任务：分身可保持无状态。
2.2.3 工具与组件装配（Tools & Components）
MCP 协议支持：通过大模型上下文协议（MCP），分身可标准化地调用外部数据源和工具。
多源工具接入：
提供商提供的云端 API
本地原生工具（如本地文件读写、终端命令）
由主 Agent 临时独立开发并注入的组件（如动态生成的 Python 函数）
2.3 分身的热更新与生命周期管理
独立部署与物理隔离：采用"分身部署"策略，将在线同步流程与离线异步流程物理隔离。主 Agent 部署在高性能集群保障实时响应，分身 Agent 部署在独立计算集群或容器中，按需横向扩容。
动态团队生命周期（Team Lifecycle）：通过类似 TeamCreate/Delete 的通信机制，主 Agent 可随时组建、销毁或替换分身团队，适配动态任务规模调整。
异常处理与无缝切换：当检测到分身执行失败或需要更新时，主 Agent 可自动重试、切换备用分身，或在后台拉取最新版本的组件/提示词，在下一次任务调度时生效，实现平滑热更新。
2.4 核心协同与通信机制
指令标准化：主 Agent 向分身发送的指令采用统一格式（包含任务 ID、参数、反馈要求），降低沟通成本。
消息传递与任务看板：使用 SendMessageTool 进行实时数据交互，使用 TaskCreate/Update 共享任务看板，实现进度实时同步与统一调度。
结果聚合与校验：分身完成任务后，主 Agent 对结果进行校验。若不符合要求，发送修正指令让分身重新执行；若符合，进行二次加工或触发下一轮任务。
第三部分：记忆系统 — 标准化抽象与智能路由
3.1 记忆组件的标准化与抽象化
由于各提供商底层技术差异巨大（云端 API / 本地 SQLite），分身系统不能直接硬编码调用。引入记忆抽象层（Memory Abstraction Layer）：

统一接口协议：定义标准 Memory Interface（save(), search(), get_profile(), update_graph()）。无论底层是 Hindsight 的知识图谱还是 Holographic 的 FTS5 搜索，对分身而言调用的都是统一的标准化方法。
MCP 协议封装：将这些记忆提供商封装为标准的 MCP Server。分身通过 MCP 协议即插即用地调用记忆服务，无需关心底层是本地文件还是云端 API。
3.2 支持的记忆提供商清单（必须全部预留适配位）
提供商	能力特征	适用场景
Hindsight	长期记忆，知识图谱 + 多策略检索	深度推理与复杂关联，长链路逻辑和海量历史档案
Holographic	本地 SQLite 事实存储，FTS5 搜索 + 信任评分，无需 API Key	隐私保护与高频本地事实查询，零延迟
Honcho	基于 AI 的跨会话用户画像建模，辩证问答 + 语义搜索	个性化交互与用户画像建模，“记住用户偏好”
Mem0	服务端 LLM 事实提取，语义搜索 + 自动去重	轻量级事实提取与去重，快速记录对话中的客观事实
OpenViking	会话管理的记忆，分层检索 + 知识浏览	结构化浏览与知识导航，构建当前任务的认知地图
RetainDB	云端记忆 API，混合搜索 + 7 种记忆类型	深度推理与复杂关联（云端方案）
Supermemory	语义长期记忆，档案回忆 + 实体提取	结构化浏览与知识导航，实体提取
3.3 基于"任务特征"的记忆路由策略
主 Agent 或分身自身的配置引擎应根据任务类型智能选配最合适的记忆提供商：

需要深度推理与复杂关联 → 装配 Hindsight 或 RetainDB
需要隐私保护与高频本地事实查询 → 装配 Holographic
需要个性化交互与用户画像建模 → 装配 Honcho
需要轻量级事实提取与去重 → 装配 Mem0
需要结构化浏览与知识导航 → 装配 OpenViking 或 Supermemory
3.4 记忆生命周期与隔离机制（两层）
工作记忆（Working Memory）：仅存在于分身的独立上下文窗口（Context Window）中。分身执行任务时，通过检索工具将相关的长期记忆（如从 Holographic 提取的几条关键事实）注入到当前上下文中。任务结束后，这部分工作记忆随分身销毁。
持久化记忆（Long-term Memory）：由记忆提供商接管。分身执行完毕后，如果产生了新的有价值的事实或用户偏好，分身会触发一个 memory_update 动作，将结构化数据写回对应的提供商（如写入 Mem0 或更新 Hindsight 的图谱）。
3.5 支持热更新与动态切换
运行时动态替换：如果主 Agent 发现当前任务从"本地代码分析"切换到了"跨月历史数据调研"，可动态调整分身的配置，将记忆组件从 Holographic 无缝切换为 RetainDB。
降级与容错机制：如果某个云端记忆提供商（如 Hindsight）的 API 暂时不可用，分身的记忆抽象层可自动降级，切换到备用的本地记忆方案（如 Holographic），保证任务不中断。
第四部分：高阶认知能力 — 规划、反思与发现学习
4.1 规划（Planning）— 从目标到执行路径的导航
任务分解与编排：主 Agent 需具备任务树分解算法，将复杂需求转化为 DAG（有向无环图）或状态机。例如，将"开发一个电商模块"拆解为需求分析、数据库设计、代码编写、测试等子任务链。
动态调整与混合调度：计划并非一成不变。主 Agent 需结合基于规则（如流程图）与基于模型（如 ReAct、Chain-of-Thought）的混合调度器，根据子任务的实时反馈动态更新执行计划，平衡计划的泛化能力与可控性。
4.2 反思与自我纠错（Reflection）— 系统的"元认知"
ReAct + 反思机制：在执行完一轮动作后，Agent 需自我评估结果质量。若发现偏离目标或工具调用失败，能够主动重试或调整策略，而非无脑报错。
双 Agent 校验（Actor-Critic 架构）：对于高风险或高复杂度任务，可设置"执行分身"与"审核分身"。一个负责生成代码或撰写报告，另一个负责审查并提出修改建议。
成本权衡：反思机制会显著增加 Token 消耗和延迟。工程化落地时，需通过小模型（如轻量级 LLM）代跑反思任务，在准确性需求与成本之间取得平衡。
4.3 发现与学习（Discovery & Learning）— 进化的引擎
记忆压缩与去重（知识沉淀）：结合记忆提供商，Agent 需具备自动压缩与去重能力。通过轻量级模型将冗长的对话历史提炼为结构化摘要（提取核心需求、关键事实、约束条件等），并剔除冗余信息，实现"永久记忆"而不爆显存。
动态工具与组件的"发现"：当主 Agent 发现现有工具无法满足需求时，除了临时编写代码，还应具备"发现"能力。通过检索内部知识库或 MCP 工具注册中心，动态发现并接入新的外部 API 或现成组件，丰富自身的"工具箱"。
强化学习与轨迹感知：在多 Agent 协作中，通过记录每个 Agent 的决策路径（Thought → Action → Observation 链路）形成轨迹日志。利用强化学习（如 PPO 算法）分析这些历史轨迹，优化未来的任务分配策略与协作效率，使整个系统越用越聪明。
4.4 多智能体博弈与共识机制（Multi-Agent Debate & Consensus）
多 Agent 辩论（Debate Mechanism）：面对没有标准答案的复杂决策（如商业策略制定、代码架构选型），主 Agent 可召唤多个具有不同系统提示词或不同记忆背景的分身。让它们针对同一问题发表观点、互相质询。通过多轮辩论收敛出最优解，极大降低单一模型的幻觉和偏见。
共识与投票机制（Voting & Consensus）：对于关键的工具调用或代码执行，引入类似 Raft 共识的机制。比如让 3 个独立的分身分别生成一段代码，然后通过一个"评审分身"进行投票，只有多数通过的结果才会被真正执行。
4.5 人机协同（Human-in-the-Loop, HITL）
置信度阈值与人工接管：主 Agent 在分发任务时，如果评估当前分身的置信度低于阈值，或遇到高风险操作（如修改生产环境数据库），应主动暂停执行，并将决策权交还给人类。
渐进式授权：系统应记录人类对 Agent 决策的修改历史。当人类多次修正 Agent 的某类决策后，系统可通过"发现与学习"机制，将这些修正固化为新的规则或记忆，从而逐步扩大 Agent 的自主权限。
第五部分：系统工程与可观测性
5.1 全链路追踪（Distributed Tracing）
为每一次用户请求生成一个全局 Trace_ID。
主 Agent 拆解任务、分身调用工具、记忆检索等所有动作，都必须挂载这个 ID。
通过类似 LangSmith 或 OpenTelemetry 的机制，清晰看到一条请求在系统内的完整流转图，以及每个节点的耗时和 Token 消耗。
5.2 多维度的评估与监控（Evaluation & Monitoring）
不仅看系统是否报错，还要监控"智能指标"：任务完成率、平均重试次数、幻觉率、记忆检索准确率。
引入一个轻量级的"评估 Agent"，定期对线上日志进行抽样打分，持续监控系统的健康度。
5.3 结构化与不可篡改的审计日志
每次系统活动、数据转换、决策节点都必须生成包含时间戳、输入输出哈希、调用栈深度的结构化日志（JSON 格式）。
建立区块链式或追加写入（Append-only）的日志存储，确保全生命周期内的操作绝对可追溯。
5.4 执行轨迹可视化（Execution Trace）
为每个任务生成独立的 execution_trace.json。
不仅记录成功路径，还要记录失败原因和重试逻辑，让系统的每一次"思考"都变得完全透明。
5.5 自解释 UI（Self-Explanatory UI）
前端界面应直接映射后端的执行轨迹。
当 Agent 执行复杂任务时，UI 上能实时展示工作流的流转节点、当前正在调用的工具、以及模型输出的中间推理过程。
第六部分：安全、合规与边界控制
6.1 沙箱隔离与权限最小化
主 Agent 临时开发的代码或调用的本地工具，必须在严格的沙箱（如 Docker 容器或 WebAssembly 环境）中运行。
必须实施基于角色的访问控制（RBAC）：
负责代码生成的分身绝对不允许拥有网络外发权限。
负责记忆写入的分身不允许执行系统命令。
6.2 输入输出护栏（Guardrails）
在主 Agent 与分身之间、分身与外部工具之间，必须加入安全过滤层。
防止提示词注入（Prompt Injection）攻击。
拦截敏感数据（PII）的意外输出。
确保系统不会执行破坏性指令（如 rm -rf）。
6.3 伦理与规则硬编码
将合规要求、伦理底线写入系统的基础规则中，形成不可突破的硬约束。
6.4 异常熔断机制
设置实时风险监测阈值，一旦检测到非法访问、恶意指令或异常行为，立即触发自动熔断，阻断风险扩散。
第七部分：资源与成本优化
7.1 动态路由与模型降级
系统必须具备成本感知能力。
主 Agent 在拆解任务时，应优先将简单子任务路由给小模型（如 7B/8B 模型）。
只有在遇到复杂推理时才调用昂贵的旗舰模型。
7.2 上下文压缩与缓存（Semantic Caching）
对于重复或高度相似的用户请求，系统应直接命中缓存返回结果。
在分身之间传递信息时，不要传递原始长文本，而是传递由小模型提取的"结构化摘要"，大幅降低通信成本。
第八部分：本地优先与端到端闭环
8.1 分层解耦与混合决策引擎
规则引擎主导：对于明确的业务逻辑（如数据清洗、格式转换、文件流转），全部采用硬编码或本地工作流（基于 Python 脚本或状态机）处理，保证 100% 确定性和零 Token 消耗。
LLM 作为兜底：仅在遇到非结构化数据处理、复杂意图识别等"规则失效"的场景时，才动态触发 LLM 介入。
8.2 本地推理引擎
采用 vLLM 或 llama.cpp 或 Ollama 等本地推理引擎，确保所有模型权重在本地运行，数据不出内存，实现真正的"物理不可逾越的信任边界"。
8.3 离线工具链
所有的感知层组件（如 PDF 解析、OCR）和工具执行层必须支持纯离线运行。
禁用任何默认的网络请求或远程注册表依赖。
8.4 声明式配置管理
所有的数值、凭证、模型路径、工具权限等，一律通过 YAML/JSON 配置文件驱动。
敏感凭证（如 API Key、数据库密码）必须通过密钥管理系统（KMS）或环境变量动态加载，严禁硬编码。
工具调用必须在配置文件中显式声明允许的路径和权限（如 allowed_paths），超出范围直接报错，而非静默失败。
8.5 全面测试与质量保障
评测即代码（Eval as Code）：将测试用例和评估基准（Benchmark）内建到 CI/CD 流水线中。每一次代码提交或模型更新，都必须通过自动化回归测试。
多维度验证：除了常规的单元测试和集成测试，还需针对 AI 系统引入专项测试：
对抗样本防御测试（鲁棒性）
偏见检测测试（公平性）
基于 LIME 等工具的可解释性验证
E2E（端到端）测试：模拟真实用户请求，从 UI 输入 → 触发工作流 → 调用本地 LLM → 写入本地记忆 → 最终 UI 输出结果，验证整个闭环在离线状态下是否完全通畅。

11.1 审核清单（9 条不可逾越的红线）
#	审核项	审核要点
1	工作流优先原则	系统是否真正做到了"确定性规则与模型混合驱动"？对于明确的业务逻辑，是否全部由本地工作流/脚本处理，LLM 是否仅作为非结构化任务的兜底备用方案？
2	本地可用性与端到端闭环	系统是否具备物理隔离的本地运行能力？在完全断网环境下，本地推理引擎（Ollama/vLLM）、本地记忆（SQLite/Holographic）和原生工具是否能支撑完整的端到端闭环？
3	极致的可观测性与可追溯性	系统是否实现了全链路追踪（Trace_ID）？所有操作、数据转换和决策是否都有带时间戳的结构化审计日志？是否采用了追加写入（Append-only）机制？
4	自解释 UI 交互	前端 UI 是否做到了自解释？是否能直观、实时地映射后端的执行轨迹，并清晰区分"工作流节点"与"LLM 兜底节点"的状态切换？
5	声明式配置与零硬编码	系统是否彻底消除了硬编码的数值、凭证及配置参数？敏感凭证是否通过环境变量或 KMS 动态注入？工具权限是否在配置中显式声明？
6	解耦设计与模块化	核心调度、分身容器、记忆抽象层、可观测性等模块是否做到了松耦合？各组件是否可以独立维护和替换？
7	安全沙箱与边界控制	分身的动态执行是否被严格限制在沙箱内？是否具备完善的输入输出护栏（Guardrails）以防止提示词注入或越权操作？
8	高阶认知闭环	系统是否独立实现了"反思与学习（Reflection & Learning）"模块？是否具备自我纠错、轨迹感知和知识沉淀的能力？
9	全面测试与质量保障	系统是否包含了自动化测试集？是否强制实现了离线状态下的 E2E（端到端）测试，以验证全链路的可用性？
11.2 审核输出要求
架构合规性审查报告：以表格形式输出上述 9 项的审核结果及证据说明。
偏离度评估：指出当前系统在哪些地方偏离了最初的设计思想，并评估偏离带来的风险。
整改建议：针对【部分通过】或【未通过】的项，给出明确的、可落地的代码级或架构级修复建议。
