## P3 具体执行任务

### 1. 定义统一的记忆抽象层接口

在 `memory/base.py` 中定义标准接口（`save()`, `search()`, `get_profile()`, `update_graph()`），确保与底层提供商解耦。

### 2. 完成 Holographic 适配器

在 `memory/holographic_adapter.py` 中实现本地 SQLite 事实存储 + FTS5 搜索 + 信任评分，无需 API Key。

### 3. 完成 Mem0 适配器

在 `memory/mem0_adapter.py` 中实现服务端 LLM 事实提取 + 语义搜索 + 自动去重。

### 4. 预留其余 5 个适配器骨架

在 `memory/hindsight_adapter.py`、`honcho_adapter.py`、`openviking_adapter.py`、`retaindb_adapter.py`、`supermemory_adapter.py` 中实现接口骨架与"需人工核实"标注。

### 5. 实现基于任务特征的记忆智能路由

在 `memory/router.py` 中编写路由分发逻辑，根据任务类型（深度推理/隐私保护/个性化画像/轻量事实/知识导航）自动选择最合适的记忆提供商。

### 6. 实现降级与容错

云端提供商不可用时自动降级到本地 Holographic。

### 7. 实现工作记忆与持久化记忆分离

工作记忆随分身销毁；持久化记忆通过 `memory_update` 写回提供商。

## 📎 设计参考：记忆系统 — 标准化抽象与智能路由（第三部分）

> **以下设计规范为 P3 阶段的核心指导原则，执行时必须严格遵循。**

### 3.1 记忆组件的标准化与抽象化

由于各提供商底层技术差异巨大（云端 API / 本地 SQLite），分身系统不能直接硬编码调用。引入记忆抽象层（Memory Abstraction Layer）：

- **统一接口协议**：定义标准 Memory Interface（`save()`, `search()`, `get_profile()`, `update_graph()`）。无论底层是 Hindsight 的知识图谱还是 Holographic 的 FTS5 搜索，对分身而言调用的都是统一的标准化方法。
- **MCP 协议封装**：将这些记忆提供商封装为标准的 MCP Server。分身通过 MCP 协议即插即用地调用记忆服务，无需关心底层是本地文件还是云端 API。

### 3.2 支持的记忆提供商清单（必须全部预留适配位）

| 提供商 | 能力特征 | 适用场景 |
|--------|---------|---------|
| Hindsight | 长期记忆，知识图谱 + 多策略检索 | 深度推理与复杂关联，长链路逻辑和海量历史档案 |
| Holographic | 本地 SQLite 事实存储，FTS5 搜索 + 信任评分，无需 API Key | 隐私保护与高频本地事实查询，零延迟 |
| Honcho | 基于 AI 的跨会话用户画像建模，辩证问答 + 语义搜索 | 个性化交互与用户画像建模，"记住用户偏好" |
| Mem0 | 服务端 LLM 事实提取，语义搜索 + 自动去重 | 轻量级事实提取与去重，快速记录对话中的客观事实 |
| OpenViking | 会话管理的记忆，分层检索 + 知识浏览 | 结构化浏览与知识导航，构建当前任务的认知地图 |
| RetainDB | 云端记忆 API，混合搜索 + 7 种记忆类型 | 深度推理与复杂关联（云端方案） |
| Supermemory | 语义长期记忆，档案回忆 + 实体提取 | 结构化浏览与知识导航，实体提取 |

### 3.3 基于"任务特征"的记忆路由策略

主 Agent 或分身自身的配置引擎应根据任务类型智能选配最合适的记忆提供商：

- 需要深度推理与复杂关联 → 装配 Hindsight 或 RetainDB
- 需要隐私保护与高频本地事实查询 → 装配 Holographic
- 需要个性化交互与用户画像建模 → 装配 Honcho
- 需要轻量级事实提取与去重 → 装配 Mem0
- 需要结构化浏览与知识导航 → 装配 OpenViking 或 Supermemory

### 3.4 记忆生命周期与隔离机制（两层）

**工作记忆（Working Memory）**：仅存在于分身的独立上下文窗口（Context Window）中。分身执行任务时，通过检索工具将相关的长期记忆（如从 Holographic 提取的几条关键事实）注入到当前上下文中。任务结束后，这部分工作记忆随分身销毁。

**持久化记忆（Long-term Memory）**：由记忆提供商接管。分身执行完毕后，如果产生了新的有价值的事实或用户偏好，分身会触发一个 `memory_update` 动作，将结构化数据写回对应的提供商（如写入 Mem0 或更新 Hindsight 的图谱）。

### 3.5 支持热更新与动态切换

**运行时动态替换**：如果主 Agent 发现当前任务从"本地代码分析"切换到了"跨月历史数据调研"，可动态调整分身的配置，将记忆组件从 Holographic 无缝切换为 RetainDB。

**降级与容错机制**：如果某个云端记忆提供商（如 Hindsight）的 API 暂时不可用，分身的记忆抽象层可自动降级，切换到备用的本地记忆方案（如 Holographic），保证任务不中断。
