## P2 核心交付物

- Subagent 生命周期管理（动态创建/执行/销毁/热更新）
- 严格的沙箱隔离与显式权限声明
- 本地原生工具封装 + MCP 协议基础接入 + 输入输出护栏

---

## P2 具体执行任务

### 1. 实现 Subagent 生命周期管理

在 `agents/subagent.py` 中编写分身的动态创建、执行、销毁逻辑，支持热更新（运行时动态替换组件、降级容错）。

### 2. 实现严格的沙箱隔离与权限声明

在 `agents/sandbox.py` 中实现基于配置的显式权限声明（如 `allowed_paths`），超出范围直接拦截报错。支持 Docker 容器或 WebAssembly 沙箱。

### 3. 封装本地原生工具

在 `agents/tools/native_tools.py` 中实现文件读写、终端命令等本地工具，受沙箱权限约束。

### 4. 实现 MCP 协议基础接入

在 `agents/tools/mcp_client.py` 中实现 MCP 客户端，支持标准化调用外部数据源和工具。

### 5. 实现输入输出护栏

在 `agents/tools/guardrails.py` 中实现安全过滤层，防止提示词注入、拦截 PII、拦截破坏性指令（如 rm -rf）。

### 6. 实现 RBAC

代码生成分身无网络外发权限；记忆写入分身无系统命令权限。

## 📎 设计参考：核心架构设计 — 主从协同与分身隔离（第二部分）

> **以下设计规范为 P2 阶段的核心指导原则，执行时必须严格遵循。**

### 2.1 两层架构

整个系统分为"统筹大脑"和"专业分身"两层，通过标准化协议通信。

**2.1.1 主 Agent（Orchestrator / 统筹大脑）**
- 核心定位：聚焦统筹，简化执行。不需要具备复杂的专业执行能力，专注于理解用户意图、任务拆解、资源调度和结果整合。
- 基础能力：仅保留最基本的核心逻辑和轻量级开发/编排功能（如基于 Python 或 TypeScript 的原生语法编排）。
- 动态生成：具备"动态分身（Dynamic Spawning）"能力，无需预先配置复杂的服务器，根据任务需求随时按需创建分身。

**2.1.2 分身 Agent（Subagent / 专业执行者）**
- 核心定位：单一职责、轻量化、用完即抛。干完任务后将结果交回主 Agent，自身不留下长期协调成本。
- 上下文隔离：采用独立的上下文窗口（如 Fork 进程模式或独立容器），避免长文本或复杂操作污染主 Agent 的核心状态。

### 2.2 分身的"全选配"模块化设计

每个分身由以下三个核心组件定义，支持按需装配：

**2.2.1 异构模型配置（Heterogeneous Models）**
- 允许为不同分身分配最适合的模型：
  - 复杂推理任务 → 思考型模型（如 o1, R1）
  - 快速搜索 → 速度型模型（如 Flash, Turbo）
  - 创意写作 → 文采型模型
- 框架需做到"提供商无关"，兼容多种模型 API。

**2.2.2 记忆与状态管理（Memory & State）**
- 支持状态持久化和快照机制。
- 耗时任务：分身可保存执行断点，中断后能从断点恢复。
- 轻量级任务：分身可保持无状态。

**2.2.3 工具与组件装配（Tools & Components）**
- MCP 协议支持：通过大模型上下文协议（MCP），分身可标准化地调用外部数据源和工具。
- 多源工具接入：
  - 提供商提供的云端 API
  - 本地原生工具（如本地文件读写、终端命令）
  - 由主 Agent 临时独立开发并注入的组件（如动态生成的 Python 函数）

### 2.3 分身的热更新与生命周期管理

- 独立部署与物理隔离：采用"分身部署"策略，将在线同步流程与离线异步流程物理隔离。主 Agent 部署在高性能集群保障实时响应，分身 Agent 部署在独立计算集群或容器中，按需横向扩容。
- 动态团队生命周期（Team Lifecycle）：通过类似 TeamCreate/Delete 的通信机制，主 Agent 可随时组建、销毁或替换分身团队，适配动态任务规模调整。
- 异常处理与无缝切换：当检测到分身执行失败或需要更新时，主 Agent 可自动重试、切换备用分身，或在后台拉取最新版本的组件/提示词，在下一次任务调度时生效，实现平滑热更新。

### 2.4 核心协同与通信机制

- 指令标准化：主 Agent 向分身发送的指令采用统一格式（包含任务 ID、参数、反馈要求），降低沟通成本。
- 消息传递与任务看板：使用 SendMessageTool 进行实时数据交互，使用 TaskCreate/Update 共享任务看板，实现进度实时同步与统一调度。
- 结果聚合与校验：分身完成任务后，主 Agent 对结果进行校验。若不符合要求，发送修正指令让分身重新执行；若符合，进行二次加工或触发下一轮任务。

## 📎 设计参考：安全、合规与边界控制（第六部分）

> **以下安全规范为 P2 阶段必须实现的硬约束，不可妥协。**

### 6.1 沙箱隔离与权限最小化

主 Agent 临时开发的代码或调用的本地工具，必须在严格的沙箱（如 Docker 容器或 WebAssembly 环境）中运行。

必须实施基于角色的访问控制（RBAC）：
- 负责代码生成的分身绝对不允许拥有网络外发权限。
- 负责记忆写入的分身不允许执行系统命令。

### 6.2 输入输出护栏（Guardrails）

在主 Agent 与分身之间、分身与外部工具之间，必须加入安全过滤层。
- 防止提示词注入（Prompt Injection）攻击。
- 拦截敏感数据（PII）的意外输出。
- 确保系统不会执行破坏性指令（如 rm -rf）。

### 6.3 伦理与规则硬编码

将合规要求、伦理底线写入系统的基础规则中，形成不可突破的硬约束。

### 6.4 异常熔断机制

设置实时风险监测阈值，一旦检测到非法访问、恶意指令或异常行为，立即触发自动熔断，阻断风险扩散。
