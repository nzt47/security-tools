## P1 核心交付物

- "工作流优先"的本地规则执行引擎（状态机流转）
- 显式集成的本地推理引擎（Ollama / vLLM / llama.cpp）
- 主 Agent 统筹逻辑骨架（任务拆解 + 混合决策）

---

## P1 具体执行任务

### 1. 实现"工作流优先"的本地规则执行引擎

在 `core/workflow_engine.py` 中编写状态机流转逻辑。要求：先尝试匹配本地确定性规则/脚本，若失败或遇到非结构化任务，再触发 LLM 兜底。

### 2. 显式集成本地推理引擎

在 `core/local_llm.py` 中实现调用本地推理引擎（Ollama 或 vLLM）的接口，确保在完全断网环境下依然能加载本地模型权重进行推理。

### 3. 实现主 Agent 统筹逻辑骨架

在 `core/orchestrator.py` 中实现任务拆解与混合决策的基础代码，支持：
- 任务树分解（DAG / 状态机）
- 基于规则 + 基于模型（ReAct、CoT）的混合调度
- 动态调整执行计划

### 4. 验证断网可用性

编写测试用例，断网状态下加载本地模型完成一次推理。

## 📎 设计参考：本地优先与端到端闭环（第八部分 8.1–8.3 节）

> **以下设计规范为 P1 阶段的核心指导原则，执行时必须严格遵循。**

### 8.1 分层解耦与混合决策引擎

**规则引擎主导**：对于明确的业务逻辑（如数据清洗、格式转换、文件流转），全部采用硬编码或本地工作流（基于 Python 脚本或状态机）处理，保证 100% 确定性和零 Token 消耗。

**LLM 作为兜底**：仅在遇到非结构化数据处理、复杂意图识别等"规则失效"的场景时，才动态触发 LLM 介入。

### 8.2 本地推理引擎

采用 vLLM 或 llama.cpp 或 Ollama 等本地推理引擎，确保所有模型权重在本地运行，数据不出内存，实现真正的"物理不可逾越的信任边界"。

### 8.3 离线工具链

所有的感知层组件（如 PDF 解析、OCR）和工具执行层必须支持纯离线运行。

禁用任何默认的网络请求或远程注册表依赖。
