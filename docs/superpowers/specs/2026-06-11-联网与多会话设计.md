# 云枢联网与多会话设计

## 概述

为云枢（Yunshu）增加两个核心能力：
1. **联网能力** — 让 LLM 能够自主调用已注册的互联网工具（web_search、web_get 等），获取实时信息
2. **多会话管理** — 支持创建/切换/管理多个独立对话会话，实现类 ChatGPT 的完整体验

## 背景

当前现状：
- `agent/web/` 已有完整互联网模块（HttpClient、SearchEngine、Scraper 等）
- DigitalLife 已注册 `web_search`、`web_get` 等 8 个联网工具
- 但 LLM 调用层（`memory/llm_service.py`）不支持 tool calling，工具对 LLM 不可见
- 聊天历史仅存于内存 `_CHAT_HISTORY` 列表，新消息不持久化到文件
- 前端只有一个对话视图，不支持多会话切换

## 1. 多会话管理系统

### 1.1 数据模型

**文件结构：**
```
data/sessions/
  ├── sessions.json              # 会话索引
  ├── {session_id}/
  │     ├── messages.jsonl       # 消息记录（追加写）
  │     └── meta.json            # 会话元数据
  └── ...
```

**sessions.json 格式：**
```json
[
  {
    "id": "sess_20260611_xxxxx",
    "title": "关于项目架构的讨论",
    "created_at": "2026-06-11T10:00:00+08:00",
    "updated_at": "2026-06-11T11:30:00+08:00",
    "message_count": 15,
    "mode": "normal"
  }
]
```

**messages.jsonl 格式：**
```jsonl
{"role":"user","content":"你好，云枢","timestamp":"..."}
{"role":"assistant","content":"你好！我是云枢...","timestamp":"...","tool_calls":null}
{"role":"user","content":"帮我搜索今天的新闻","timestamp":"..."}
{"role":"assistant","content":"好的，我来搜索...","timestamp":"...","tool_calls":[{"name":"web_search","arguments":{"query":"今日新闻"},"result":"..."}]}
```

**meta.json 格式：**
```json
{
  "id": "sess_...",
  "title": "...",
  "mode": "normal",
  "llm_config": {"provider": "openai", "model": "gpt-4"},
  "summary": "关于项目架构的深入讨论...",
  "expire_at": null
}
```

### 1.2 SessionManager 类

```python
class SessionManager:
    """多会话管理器"""

    def __init__(self, sessions_dir: str = "./data/sessions"):
        self._sessions_dir = Path(sessions_dir)
        self._current_id: str | None = None  # 当前活跃会话ID
        self._lock = threading.Lock()

    def list_sessions(self, sort_by="updated_at", limit=50) -> list[SessionInfo]

    def get_session(self, session_id: str) -> SessionInfo | None

    def rename_session(self, session_id: str, title: str) -> bool

    # ── 当前会话管理 ──
    def set_current(self, session_id: str) -> bool
    def get_current(self) -> SessionInfo | None
    def get_current_id(self) -> str | None

    # ── 消息操作 ──
    def add_message(self, session_id: str, role: str, content: str,
                    tool_calls: list | None = None) -> dict
    def get_messages(self, session_id: str, limit: int = 50,
                     offset: int = 0) -> list[dict]
    def clear_messages(self, session_id: str) -> bool

    # ── 标题自动生成 ──
    def _auto_title(self, session_id: str) -> str
    def _update_timestamp(self, session_id: str)
```

### 1.3 线程安全与性能

- 每个会话的写入使用 `threading.Lock()` 保护
- `sessions.json` 使用独立锁，写后立即刷盘
- 消息文件使用追加写入（O(1)），不加载全量到内存
- 会话列表按 `updated_at` 降序缓存，每次变更后刷新

## 2. 联网能力 — ToolCallingService

### 2.1 核心流程

```
用户输入
  │
  ▼
ToolCallingService.chat(messages, system_prompt, tools_whitelist=None)
  │
  ├── Round 1:
  │     ├── 构建 tool_definitions（从 tools 注册表 + whitelist 过滤）
  │     ├── 调用 LLM API（带 tools 参数）
  │     │
  │     ├── LLM 返回文本 → goto END
  │     └── LLM 返回 tool_calls →
  │           ├── 对每个 tool_call：
  │           │     ├── 从注册表查找 handler
  │           │     ├── 设置超时（默认 60s）
  │           │     ├── 执行工具
  │           │     └── 记录结果
  │           └── 将 tool_result 作为新消息追加
  │
  ├── Round 2 (如有工具调用):
  │     └── 再次调用 LLM（携带工具结果）
  │
  └── 达到 max_rounds（默认 5）或 LLM 返回文本 → END
        │
        END → 返回 LLM 最终回复文本
```

### 2.2 ToolCallingService 类

```python
class ToolCallingService:
    """LLM 工具调用编排引擎"""

    def __init__(self, llm_service: LLMService, max_rounds: int = 5,
                 tool_timeout: int = 60):
        self._llm = llm_service
        self._max_rounds = max_rounds
        self._tool_timeout = tool_timeout

    def chat(self, messages: list[dict], system_prompt: str = "",
             max_tokens: int = 1024, temperature: float = 0.7,
             tools_whitelist: list[str] | None = None) -> str:
        """带工具调用的对话

        Args:
            messages: 对话消息列表
            system_prompt: 系统提示词
            max_tokens: 最大生成 Token 数
            temperature: 生成温度
            tools_whitelist: 允许调用的工具名称列表（None 表示全部可用工具）

        Returns:
            LLM 生成的最终回复文本
        """

    def _build_tool_defs(self, whitelist: list[str] | None) -> list[dict]:
        """将 tools 注册表转换为 OpenAI/Anthropic 格式的 tool definitions"""

    def _execute_tool(self, tool_name: str, arguments: dict) -> dict:
        """执行单个工具并返回结果"""

    def _format_tool_result(self, tool_call_id: str, tool_name: str,
                            result: Any) -> dict:
        """将工具执行结果格式化为可注入消息"""
```

### 2.3 工具定义转换

当前工具注册表格式：
```python
_registry = {
    "web_search": {
        "name": "web_search",
        "description": "搜索互联网信息...",
        "handler": <function>
    }
}
```

转换为 OpenAI tool format：
```python
{
    "type": "function",
    "function": {
        "name": "web_search",
        "description": "搜索互联网信息（默认使用 DuckDuckGo，无需 API Key）",
        "parameters": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "搜索关键词"},
                "num_results": {"type": "integer", "description": "返回结果数，默认 10"},
                "engine": {"type": "string", "description": "搜索引擎，可选 duckduckgo/tavily/bing/google"}
            },
            "required": ["query"]
        }
    }
}
```

### 2.4 工具参数字段标注

为每个现有工具补充参数 schema。标注方式：在 `register()` 时添加 `schema` 元数据：

```python
@register("web_search", "搜索互联网信息", schema={
    "type": "object",
    "properties": {
        "query": {"type": "string", "description": "搜索关键词"},
        "num_results": {"type": "integer", "description": "返回结果数"},
    },
    "required": ["query"]
})
def _web_search(**kwargs):
    ...
```

### 2.5 需要标注 schema 的工具清单

| 工具名 | 描述 | 关键参数 |
|--------|------|----------|
| web_search | 搜索互联网 | query, num_results, engine, page |
| web_get | 发送 HTTP GET 请求 | url, timeout, headers |
| web_post | 发送 HTTP POST 请求 | url, data, json_data, timeout |
| web_xpath | 使用 XPath 提取信息 | url/ html, expression |
| web_css | 使用 CSS 选择器提取信息 | url/ html, selector, attr |
| web_download | 下载文件 | url, filepath |
| web_batch | 批量请求 | urls |
| web_clean_data | 清洗数据 | text/ items |
| check_health | 检查身体状态 | 无参数 |
| get_status | 获取完整状态 | 无参数 |
| search_memory | 搜索记忆 | query |
| read_file | 读取文件 | path, encoding, max_size_mb |
| write_file | 写入文件 | path, content, encoding |

| search_files | 搜索文件 | pattern, path |

### 2.6 异常处理

| 场景 | 处理方式 |
|------|---------|
| 工具执行异常 | 将错误信息包装为 tool result 返回给 LLM，由 LLM 决定如何回应 |
| 工具执行超时 (60s) | 中断工具执行，返回 "工具执行超时" 提示给 LLM |
| LLM API 调用失败 | 降级为普通文本对话（不带 tools 参数重试） |
| 工具循环超过 max_rounds | 强制终止循环，返回当前最后一条 LLM 回复 |
| 未知工具名称 | 返回 "未知工具" 错误给 LLM |

### 2.7 与现有系统的集成

在 `DigitalLife._call_llm()` 中替换原来的 `self._llm.chat()` 调用：

```python
# DigitalLife._call_llm 修改点：

# 原代码：
# response = self._llm.chat(messages, system_prompt=system_prompt, ...)

# 新代码：
if self._tool_calling_service:
    response = self._tool_calling_service.chat(
        messages,
        system_prompt=system_prompt,
        max_tokens=1024,
        temperature=0.7,
    )
else:
    response = self._llm.chat(messages, system_prompt=system_prompt, ...)
```

同时需要在 DigitalLife 初始化时创建 `ToolCallingService`：

```python
# DigitalLife.__init__ 新增：
from agent.tools import list_tools, call as call_tool
from .tool_calling import ToolCallingService

# ... existing init ...
tc_service_cfg = self._config.get("tool_calling", {})
self._tool_calling_service = ToolCallingService(
    llm_service=self._llm,
    max_rounds=tc_service_cfg.get("max_rounds", 5),
    tool_timeout=tc_service_cfg.get("tool_timeout", 60),
)
```

## 4. 集成方案

### DigitalLife 变更点

1. `__init__` — 新增 `ToolCallingService` 初始化
2. `_call_llm` — 替换 LLM 调用为 ToolCallingService.chat()
3. `_chat_v2` — 同样替换 V2 路径的 LLM 调用
4. 向后兼容 — 当 ToolCallingService 不可用时，回退为原始 LLM 调用

### app_server.py 变更点

1. 新增 `SessionManager` 全局实例
2. 新增 `/api/sessions/*` 路由组
3. 修改 `/api/chat` — 接受 session 参数，将消息存入当前会话
4. 修改 `/api/history` — 读取当前会话的消息
5. 修改 `/api/clear` — 清空当前会话
6. 移除旧的文件加载逻辑 `_load_chat_history_from_file()`
7. 启动时自动创建默认会话

### 前端变更点

1. 新增会话侧边栏（HTML + CSS + JS）
2. 增强聊天区域支持多会话切换
3. 增强历史会话视图支持会话分组

## 5. 向后兼容

- 服务器启动时自动创建默认会话，旧 API 行为不变
- `_CHAT_HISTORY` 保留为当前会话的缓存，逐步迁移到 SessionManager
- ToolCallingService 降级：当 LLM 不支持 tool calling 时自动回退为纯文本模式
- 所有新增功能可通过配置开关禁用（enabled: false）

## 6. 性能考虑

- SessionManager 写入使用锁 + 追加写，不影响主线程
- 消息文件使用 JSONL 格式，支持高效追加和截断读取
- 会话列表缓存，写入后失效
- 工具调用超时控制，避免长时间阻塞
- 最大会话数量限制（默认 100），防止磁盘膨胀

## 7. 安全考虑

- 工具白名单/黑名单机制，管理员可限制 LLM 可调用的工具
- 文件路径工具（read_file/write_file）默认限定在 workspace 目录
- web 请求超时控制，防止 SSRF 类攻击
- 会话 ID 使用不可猜测的 UUID
- 敏感配置（API Key）不写入会话 meta
