### P0-001 详细分析

**缺陷描述：** `agent/cognitive/critic.py` 行 138-229 存在 3 处降级逻辑，当 Critic 服务不可用、熔断器打开或评估异常时，强制返回 `EvaluationResult(overall_score=80, passed=True)`。测试 `test_evaluate_fails_threshold` 期望低质量响应得到 `passed=False` 和 `overall_score < 70`，但降级逻辑覆盖了真实评估结果。

**影响范围：**
- 直接影响：5 个测试失败
- 间接影响：质量门禁可能放过低质量响应，影响系统输出质量
- 潜在风险：降级期间所有响应都被标记为"通过"，无法发现真实的质量回退

**修复方案：**
- 方案 A（推荐）：在测试中 mock Critic 服务为可用状态，绕过降级逻辑
- 方案 B：更新测试期望为降级行为（`passed=True`, `score=80`），并增加降级场景的独立测试
- 方案 C：修改生产代码，降级时返回 `passed=None`（未知）而非 `passed=True`

### P1-001 详细分析

**缺陷描述：** `agent/task_scheduler.py` 的任务结构从扁平的 `{"type": "cron", "day_of_week": ..., "hour": ...}` 变更为嵌套的 `{"type": "python_func", "cron": {"day_of_week": ..., "hour": ...}}`，但 `test_task_scheduler.py` 中 23 个测试仍使用旧结构。此外存在：硬编码 Windows 路径、调用不存在的 `start()` 方法、日志次数断言不匹配等问题。

**影响范围：**
- 直接影响：23 个测试失败
- 间接影响：任务调度功能无法通过自动化测试验证，存在回归风险
- 涉及子问题：
  - 6+ 个测试使用旧任务结构（cron/interval → python_func）
  - 1 个硬编码 `cwd=r"C:\Users\Administrator\agent"`
  - 1 个调用不存在的 `scheduler.start()`（应为 `start_daemon()`）
  - 1 个 `call_count >= 2` 断言（生产只有 1 次调用）

### P1-002 详细分析

**缺陷描述：** `agent/utils/sensitive_data_filter.py` 行 99 定义 `REDACTED_VALUE = "********"`（8 星），但 `test_config_secure.py` 和 `test_log_system_safe_logger.py` 中 6 个测试期望 `"***"`（3 星）。脱敏值不一致可能导致安全审计误判。

**影响范围：**
- 直接影响：6 个测试失败（4 个 config_secure + 2 个 log_system_safe_logger）
- 间接影响：脱敏策略变更未同步测试，可能掩盖脱敏失效

### P1-003 详细分析

**缺陷描述：** `agent/p6_snapshot.py` 行 838-928 的 `_restore_body_sensor`、`_restore_behavior`、`_restore_permission` 方法使用 `hasattr()` 检查字段存在性。Python 的 `hasattr()` 会吞掉 property 访问时抛出的异常并返回 False，导致异常测试用例无法触发 except 分支，函数静默返回 True 而非期望的 False。

**影响范围：**
- 直接影响：4 个测试失败（3 个 property 异常 + 1 个 PicklingError）
- 间接影响：快照恢复的异常路径可能静默成功，导致状态不一致

### P1-004 详细分析

**缺陷描述：** 全项目覆盖率 31.93%，远低于 40% 阈值。148 个文件覆盖率 0-20%（占总数 56.5%）。sensor 模块覆盖率仅 9.18%。

**影响范围：** 大量代码无测试保护，重构和修改风险高
