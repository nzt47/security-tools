窗口监控功能 — 方案对比

在记忆管理模块中扩展 OS 级窗口活动监控

方案 A：窗口传感器 + 黑盒日志集成（推荐）

架构：复用现有传感器 + 黑盒日志 + 记忆面板扩展

  + 监控开关/参数配置

优点

  • 最大程度复用现有 BlackBox 日志和 MemoryManager 架构
  • 窗口事件自然融入日志统计系统，无需额外存储
  • 改动量适中，2 个新文件 + 少量修改
  • 与现有记忆压缩/摘要机制统一

缺点

  • BlackBox JSONL 格式对时序查询不够高效
  • 统计分析需全量扫描日志文件

方案 B：专用窗口监控服务 + SQLite 存储

优点

  • SQLite 支持高效的时间范围查询和聚合
  • 独立的服务进程，与传感器系统解耦
  • 更丰富的统计能力

缺点

  • 引入新依赖（sqlite3 虽内置但增加复杂度）
  • 需要独立的存储管理、迁移、清理逻辑
  • 与现有记忆管理体系割裂，概念不统一
  • 改动量大，约 5-6 个新文件

方案 C：轻量内存缓存 + 纯前端展示

优点

  • 实现最快，无持久化逻辑
  • 零存储开销

缺点

  • 服务重启后数据全部丢失
  • 无法做历史趋势分析
  • 内存缓存有上限
  • 与"记忆管理"理念不符（记忆应该持久）
