```
   ├── cpu_sensor.py          ← CPU 传感器（温度/频率/负载/电压/风扇）

   ├── network_sensor.py      ← 网络传感器（延迟/速率/WiFi/连接数）

   ├── board_sensor.py        ← 主板传感器（型号/BIOS/芯片组/接口）
   ├── port_sensor.py         ← 端口传感器（USB/COM/LPT/音频/显示）
   ├── peripheral_sensor.py   ← 外设传感器（鼠标/键盘/显示器/游戏杆）
   ├── process_sensor.py      ← 进程传感器（负载/线程/句柄/内存）
   ├── environment_sensor.py  ← 环境传感器（Python/OS/系统时区/语言）
   ├── system_sensor.py       ← 系统传感器（显示/音频/安全/更新/日志）
   ├── behavior_sensor.py     ← 行为传感器（CPU/IO/内存行为模式分析）
   ├── hardware_file_sensor.py← 硬件文件传感器（磁盘/文件管理/快捷方式）

   ├── change_detector.py     ← 变更检测（快照对比：设备/分区/进程/服务/
   │                              注册表/环境变量）
   ├── event_monitor.py       ← 硬件事件实时监听（USB 插拔/设备变更）
   ├── file_watcher.py        ← 文件系统监控（配置文件变更）
   │
   ├── hardware_blueprint.py  ← 硬件蓝图（89 项：CPU/GPU/内存/主板/存储/网络/
   │                              音频/外设/接口的清单检测）
   ├── file_blueprint.py      ← 文件蓝图（27 项：系统盘/分区/文件系统/
   │                              系统目录/文件功能）
   ├── software_blueprint.py  ← 软件蓝图（48 项：OS/运行时/包管理器/已安装软件/
   │                              服务/驱动/安全/虚拟化/浏览器/开发工具）
   │
   ├── counter_reader.py      ← Windows 性能计数器读取
   ├── file_watcher.py        ← 文件变更监控器

```

### 核心设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| Web 框架 | Flask, threaded=False | WMI/COM 需要主线程初始化，单线程避免子线程 COM 问题 |
| 数据架构 | 启动预采集 + 内存缓存 | 避免每次请求实时采集（90s+），全量缓存和读取 |
| 类别系统 | Category 枚举 + _NAME_TO_CATEGORY 映射 | 桥接传感器类与前端 API |
| 蓝图系统 | 预定义清单 + 逐项检测 | 结构化展示"系统已知什么/未知什么" |

## 二、传感器系统

### 2.1 数据模型 ([sensor_reading.py](sensor/sensor_reading.py))

每条传感器读数都是一个 `SensorReading` 对象：

```python
SensorReading(
    sensor_name="cpu_temperature",       # 唯一标识
    value=65.3,                          # 数值
    unit="°C",                           # 单位
    description="CPU 温度（我的大脑温度）", # 中文描述（第一人称）
    category=Category.CPU,               # 类别
    severity=Severity.NORMAL,            # 严重级别
    metadata={                           # 元数据
        "thresholds": {...},
        "tags": ["硬件感知", "内部感知", "动态运行", ...]  # 8 维度标签
    }
)
```

### 2.2 Category 枚举

```
CPU, GPU, MEMORY, DISK, BATTERY, NETWORK, BOARD, CHASSIS,
PORT, PERIPHERAL, PROCESS, FILE, ENVIRONMENT, ACTIVITY,
CHANGE, SYSTEM, DISPLAY, AUDIO
```

### 2.3 Severity 级别

- `NORMAL` — 正常（绿色）
- `WARNING` — 警告（黄色）
- `CRITICAL` — 危险（红色）

### 2.4 传感器统计（预采集数据量）

| 类别 | 读数数 | 主要检测内容 |
|------|--------|-------------|
| cpu | ~39 | 温度/频率/负载/电压/风扇/功耗 |
| gpu | ~30 | 温度/显存/编码器/帧率/风扇 |
| memory | ~15-20 | 使用率/已用/可用/提交/缓存 |
| disk | ~25 | 各分区容量/IO/健康度/温度/SMART |
| battery | ~0-8 | 电量/状态/损耗/循环（台式机 0） |
| network | ~100 | 接口/延迟/速率/WiFi/ARP/TCP |
| board | ~21 | 型号/BIOS/芯片组/温度/接口状态 |
| chassis | ~5 | 温度/风扇/电源/开盖 |
| port | ~74 | USB 2.0/3.0/Type-C/COM/LPT/显示/音频 |
| peripheral | ~34 | 鼠标/键盘/显示器/触摸板/游戏杆 |
| process | ~89 | Top 进程 CPU/内存/线程/句柄 |
| environment | ~52 | Python/OS/时区/语言/编码 |
| file | ~54 | 磁盘使用/文件统计/长路径/重复文件 |
| activity | ~64 | CPU/IO/内存行为模式 |
| system | ~60 | 显示/音频/安全/更新/日志/证书 |
| change | 1+ | 变更事件（基准建立后报告差异） |

## 三、三套蓝图系统

蓝图是"已知清单"——不追求采集实时数值，而是回答"系统中有什么"。

### 3.1 [硬件蓝图](sensor/hardware_blueprint.py) — 89 项

按数据来源分为 4 类，前端按 `data_origin` 分组展示：

| data_origin | 说明 | 检测方式 |
|-------------|------|----------|
| software | 从 WMI/系统API/注册表读取 | `✅ 软件检测` |
| direct | 从硬件直读（hwmon/LibreHardwareMonitor） | `🔬 硬件直读` |
| inference | 通过软件行为推断（如 CPU 是否支持虚拟化） | `⚡ 推断` |
| manual |需人工检查（如机箱内灰尘、物理开关） | `🔧 需人工检查` |

覆盖范围：CPU、GPU、内存插槽、主板接口、存储设备、网络适配器、音频设备、外设接口、系统固件。

### 3.2 [文件蓝图](sensor/file_blueprint.py) — 27 项

按检测方法分组：软件检测 / 推断 / 需人工检查

- 存储卷：系统盘、NTFS/ReFS/FAT32 文件系统
- 分区：硬盘分区数、光驱/网络映射/BitLocker/内存盘
- 系统目录：Windows、Program Files、Users、AppData、Temp、Desktop、Documents、Downloads
- 特殊目录：System32、ProgramData、Public、Common AppData
- 文件功能：卷影复制、压缩支持、磁盘配额、文件变更监测、实时同步状态

### 3.3 [软件蓝图](sensor/software_blueprint.py) — 48 项

- 操作系统：版本、位数、安装日期、激活状态、体验指数、补丁等级
- 运行时：Python、Node.js、Java、.NET、.NET Core、PowerShell、Go、Rust、Git、Docker、WSL
- 包管理器：pip、npm、Chocolatey、Scoop、winget
- 安装软件：应用程序数、Store 应用、开机自启、服务数、驱动数、COM 组件
- 安全：Defender、第三方杀毒、防火墙状态
- 虚拟化：Hyper-V、VirtualBox、VMware、Docker Desktop
- 浏览器：Edge、Chrome、Firefox
- 开发工具：Visual Studio、VS Code、Windows Terminal
- 语言/区域：系统语言、时区、输入法
- 环境变量：PATH 目录数、JAVA_HOME、PYTHON_HOME

检测策略（已修复）：注册表 App Paths → 常见安装路径 → PATH/subprocess 三重回退

## 四、8 维度标签系统

每条传感器读数被打上 8 个维度的语义标签，前端支持按任意维度组合筛选。

| 维度 | key | 可能值 |
|------|-----|--------|
| 目标域 | domain | 硬件感知、软件感知、行为感知、环境感知 |
| 内外方位 | locus | 内部感知、外部感知、边界感知 |
| 动静属性 | temporal | 静态配置、动态运行、增量变化 |
| 采集方式 | method | 主动探测、被动监听、系统查询、对比检测 |
| 感知层次 | layer | 物理层、系统层、应用层 |
| 功能角色 | role | 基础生存、性能监控、安全防护、社交通信、环境适应 |
| 数据特征 | datatype | 数值量、状态量、事件量、配置量 |
| 可干预性 | control | 仅可观测、可配置 |

标签分配定义在 [tags.py](sensor/tags.py) 的 `CATEGORY_TAGS` + `_apply_tags()` 方法。

前端维度筛选 API: `GET /api/filter?tags=硬件感知,内部感知`

## 五、变更检测系统

[change_detector.py](sensor/change_detector.py) 采用**快照对比**机制：
1. 首次 collect() 建立基准快照
2. 后续 collect() 对比差异
3. 差异 → SensorReading 事件

### 快照内容

| 快照项 | 方式 | 数据量 |
|--------|------|--------|
| 设备列表 | WMI Win32_PnPEntity | 中等 |
| 磁盘分区 | psutil.disk_partitions | 小 |
| 进程列表 | psutil.process_iter | 大 |
| 服务列表 | WMI Win32_Service | 中等 |
| 系统信息 | platform 模块 | 小 |
| 注册表 | winreg 读取 7 条关键路径 | 小～中 |
| 环境变量 | os.environ | 小 |

### 注册表监控路径

```
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce
HKCU\Software\Microsoft\Windows\CurrentVersion\Run
HKCU\Software\Microsoft\Windows\CurrentVersion\RunOnce
HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion        (关键字段)
HKCU\Environment                                          (用户环境变量持久值)
HKLM\...\Session Manager\Environment                      (系统环境变量)
HKLM\...\Session Manager                                   (BootExecute)
HKLM\...\Policies\System                                   (UAC 策略)
```

### 配置文件监控

FileWatcher 监控 5 个目录，按扩展名过滤：

| 目录 | 说明 |
|------|------|
| `System32\drivers\etc` | hosts, networks 等网络配置 |
| `System32\GroupPolicy` | 组策略文件 |
| `%APPDATA%\...\Startup` | 用户启动项 |
| `%LOCALAPPDATA%\Microsoft\Windows` | Windows 用户配置 |
| `%LOCALAPPDATA%\Programs\Common` | 公共程序配置 |

## 六、Web 服务器

### 启动流程

1. `_init_cache()` 在主线程同步调用
2. 创建 BodySensor → 初始化所有传感器
3. 逐个类别调用 `sensor.collect()` 预采集
4. 调用 `change_detector.collect()` 建立基准快照
5. 采集 3 个蓝图
6. 存入 `_CACHE` 字典，启动 Flask

启动耗时：约 90 秒（主要瓶颈：WMI 查询 + subprocess 检测）

### API 端点

| 路径 | 说明 |
|------|------|
| `GET /` | 前端页面 |
| `GET /api/health` | 快速健康指标 |
| `GET /api/sensors` | 传感器列表 |
| `GET /api/category/<cat>` | 按类别获取读数 |
| `GET /api/blueprint` | 硬件蓝图 |
| `GET /api/file-blueprint` | 文件蓝图 |
| `GET /api/software-blueprint` | 软件蓝图 |
| `GET /api/tag-dimensions` | 8 维度定义 |
| `GET /api/filter?tags=...` | 按标签筛选读数 |

### 前端功能

- 快速健康卡片（CPU/内存/磁盘/温度概览）
- 8 维度筛选面板（点击标签组合筛选）
- 传感器列表（跳转到对应类别）
- 读数面板（显示筛选/选中类别的读数）
- 三个蓝图面板（按 method/origin 分组展示）
- 浮动刷新按钮 + 更新时间显示

## 七、已知限制

1. **启动缓慢**（~90s）— WMI 查询和 subprocess 调用在主线程顺序执行，大量超时等待。可考虑：懒加载、异步预采集、缓存持久化。
2. **WMI COM 线程模型** — `threaded=False` 限制 Flask 只能单线程处理。可迁移到异步框架。
3. **部分传感器空数据** — battery 对台式机返回 0 条；部分 WMI 类（Win32_WinSAT、Win32_WindowsProductActivation）可能不可用。
4. **注册表监控范围有限** — 仅覆盖 9 条关键路径，非全量。
5. **无数据库持久化** — 所有数据在内存中，服务重启即丢失。change_log 有 JSON 文件持久化但仅限变更事件。
6. **跨平台不完整** — 大量 Windows 特定的 WMI/winreg 调用，Linux/macOS 检测分支功能不完备。

## 八、阶段二接口依赖

阶段二的 `PromptInjector` 需要从阶段一获取的数据接口：

```python
# 1. 获取所有传感器读数（用于翻译和摘要）
readings = _CACHE["readings"]  # Dict[str, List[dict]]
# 每条 dict 包含: sensor_name, value, unit, description, category, severity, tags

# 2. 获取健康指标（用于快速状态判断）
health = _CACHE["health"]  # List[dict]

# 3. 直接访问 BodySensor 实例
from sensor import BodySensor
bs = BodySensor(...)
bs.collect_quick()     # 快速健康
bs.collect_all()       # 全量采集（慢）
```

建议 PromptInjector 接收的数据格式（与 SensorReading.to_dict() 一致）：

```python
{
    "sensor_name": "cpu_temperature",
    "value": 65.3,
    "unit": "°C",
    "description": "CPU 温度（我的大脑温度）",
    "category": "cpu",
    "severity": "normal",
    "tags": ["硬件感知", "内部感知", "动态运行", ...]
}
```
